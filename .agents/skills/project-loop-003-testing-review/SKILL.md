# /Project_loop_003_阶段测试审查

GpuRelay 项目级阶段测试与审查编排器。在每一阶段/任务完成开发后，强制按照"全量回归 → QA 验证 → A/B 对比 → 代码审查 → 测试报告"五步流程执行，确保质量达标。

执行原则：**先测试后审查，先数据后结论，先证据后放行。** 任何未经验证的"完成"声明等同于违规。

## 🔴 只读红线（最高优先级，不可违反）

本技能是**纯检查/评估/报告**技能，**绝对禁止修改任何代码文件**。

| 允许 ✅ | 禁止 ❌ |
|---------|---------|
| 运行测试命令（pytest、python -c 等） | 修改 `.py` 文件 |
| 读取代码、文档、配置文件 | 修改 `.json`、`.yaml`、`.md` 文件 |
| 输出测试结果、QA 表格、A/B 对比 | 修改 `pyproject.toml`、`requirements.txt` |
| 生成测试报告（`docs/test-reports/`） | 执行 `git commit`、`git push` |
| 列出发现的问题和建议 | 执行 `pip install`、环境变更 |
| 执行等价验证脚本（一次性验证） | 调用 `Edit`/`Write` 工具修改项目文件 |

**红线规则**：
- 发现测试失败 → **报告失败清单和根因分析**，不得修改代码去"修复"测试
- 发现代码问题 → **输出审查发现表**，不得直接修改源码
- 发现文档不一致 → **记录到报告**，不得修改文档
- 修复工作由 `Project_loop_000_工作loop约束` 的 Phase 6 或其他开发技能负责

**违规判定**：任何对项目代码/配置/文档文件的修改操作（Edit/Write）视为违规，等同于测试造假。

## 适用场景（强制触发）

| 触发条件 | 说明 |
|---|---|
| 阶段开发完成 | 用户声明某阶段/任务完成，要求审核 |
| 提交前验证 | 代码提交前，需确认所有测试通过 |
| 代码审查请求 | 用户要求对某阶段代码进行审查 |
| 修复后验证 | 代码审查问题修复后，重新验证 |
| 新环境部署 | 新硬件/新 CUDA 版本环境下的验证 |

## 五步流程（不可跳过）

```text
Step 1  全量回归测试    → pytest tests/ -v，确认 0 failed + 0 skipped
Step 2  QA 用例验证     → 按 QA 文档逐条执行，记录通过率
Step 3  A/B 性能对比    → 按 A/B 文档执行性能对比，记录阈值达标情况
Step 4  代码审查        → 调用 code-reviewer 技能，审查代码质量
Step 5  生成测试报告    → 汇总全部数据，输出到 docs/test-reports/
```

## Step 1：全量回归测试

**目标**：确认所有现有测试通过，无回归。

**执行命令**：

```bash
cd /home/ubuntu/codes/GpuRelay
.venv/bin/python -m pytest tests/ -v --tb=short
```

**验收标准**：
- 全部测试通过（0 failed）
- 极少数 skip 允许（如 cupy 未安装时 CUDA 测试跳过）
- 总耗时记录在案

**额外检查**：
- 压力测试必须通过（含 `test_stress` 中的 10k 帧测试）
- 记录 Top 10 最慢测试及其耗时

## Step 2：QA 用例验证

**目标**：按 QA 文档逐条验证功能正确性。

**执行方式**：

1. 读取 `docs/qa/` 中对应阶段的 QA 文档
2. 逐条执行 QA 文档中的验证命令
3. 如 QA 文档中标注"手动验证"，编写自动化验证脚本
4. 记录每个用例的通过/失败/跳过状态

**验收标准**：
- 正常用例通过率 100%
- 边界用例通过率 100%
- 异常用例按预期行为（如抛出指定异常）
- 回归范围中的全部测试通过

**QA 结果模板**：

```markdown
| 用例 ID | 描述 | 预期 | 实际 | 状态 |
|---------|------|------|------|------|
| N001 | ... | ... | ... | ✅/❌ |
| B001 | ... | ... | ... | ✅/❌ |
| E001 | ... | ... | ... | ✅/❌ |
```

## Step 3：A/B 性能对比

**目标**：验证性能指标达到 A/B 文档定义的阈值。

**执行方式**：

1. 读取 `docs/ab/` 中对应阶段的 A/B 文档
2. 按文档定义的采样方式执行对比测试
3. 记录 A 组（基线）和 B 组（变体）的指标
4. 判定是否达到阈值

**A/B 结果模板**：

```markdown
| 指标 | A 组 | B 组 | 阈值 | 实际 | 判定 |
|------|------|------|------|------|------|
| ... | ... | ... | ... | ... | ✅/❌ |
```

**验收标准**：
- A/B 文档中定义的全部指标达到阈值
- 如无可比维度（纯配置/文档任务），执行等价验证

**注意事项**：
- 预热 5 次以上再开始计时
- 取 100 次平均值
- 确保对比的 A/B 组在同一环境下执行

## Step 4：代码审查

**目标**：审查代码质量、功能完成度、规范合规性。

**执行方式**：

1. 调用 `code-reviewer` 技能
2. 审查变更范围内的所有文件
3. 按以下维度检查：
   - **正确性**：逻辑、边界、并发、错误路径
   - **设计**：职责单一、命名、API 稳定性
   - **安全**：注入、鉴权、敏感数据
   - **测试**：关键路径是否覆盖
   - **规范**：中文 docstring、中文日志、包 README

**审查结果模板**：

```markdown
| # | 严重度 | 问题 | 位置 | 建议 |
|---|--------|------|------|------|
| 1 | 🔴/🟡/🟢 | ... | ... | ... |
```

## Step 5：生成测试报告

**目标**：汇总全部测试数据，输出结构化报告。

**报告文件路径**：`docs/test-reports/YYYY-MM-DD-<phase>-test-report.md`

**报告必含章节**：

1. 测试环境（OS、Python、CuPy、CUDA、GPU、Git 提交）
2. 测试总览（文件数、用例数、通过/失败/跳过、总耗时、Top 10 慢测试）
3. 按测试文件统计
4. 压力测试明细（如适用）
5. 性能测试明细（如适用）
6. QA 测试汇总
7. A/B 测试汇总
8. 代码审查发现
9. 按阶段/任务统计
10. 结论与准予放行判定

**验收标准**：
- 报告包含全部 10 个章节（N/A 的标注 N/A）
- 测试数据真实可复现
- 结论明确（准予放行 / 需修复后重测）

## 闭环规则

```
Step 1 失败 → 停止，记录失败清单和根因，输出到报告
Step 2 失败 → 记录失败用例和根因分析，输出到报告
Step 3 不达标 → 记录不达标指标，评估是否可接受，输出到报告
Step 4 发现 🔴 → 记录审查发现，输出到报告
Step 5 → 报告生成后任务完成
```

**不可跳过**：任何 Step 的证据必须真实可查，不得编造数据。

**修复闭环由其他技能负责**：本技能只报告问题，不执行修复。修复工作应进入 `Project_loop_000_工作loop约束` 的 Phase 6（问题修复循环）。

## 与 Project_loop_000 的关系

本技能是 `Project_loop_000_工作loop约束` 中 Phase 5（测试验证）的详细执行器。`Project_loop_000` 定义"必须测试"，本技能定义"怎么测试、测什么、怎么判定达标"。本技能也覆盖 Phase 4 的代码审查部分。
